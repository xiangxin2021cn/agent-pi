# Agent Pi 1.1.3 MoA And Self-Correction Research

Status: research and implementation proposal  
Date: 2026-06-30  
Updated: 2026-07-01 with Hermes Agent deep-dive  
Scope: improve agent artifact quality and automatic self-correction

## Bottom Line

Agent Pi can adopt a MoA-like multi-model pattern, but the safest first version should not let several models concurrently operate on files or tools. The practical 1.1.3 direction is:

1. Keep one main executor responsible for side effects.
2. Add a provider-neutral quality orchestration layer around the existing Goal Loop.
3. Use multiple cheaper models as read-only proposers, reviewers, specialists, and aggregators.
4. Let the existing Goal Loop decide whether the task is complete, needs another internal improvement turn, or needs human review.

This is feasible in this codebase because the session layer already owns task completion state and hidden retry behavior, while agent backends already expose `queryLlm()` / `runMiniCompletion()` for secondary LLM calls.

## External Research

### Mixture of Agents

Together AI's MoA paper, "Mixture-of-Agents Enhances Large Language Model Capabilities", reports that layered cooperation among multiple LLM agents can outperform individual open-source models on AlpacaEval 2.0, MT-Bench, and FLASK. The key pattern is not a single router choosing one model. It is a multi-layer workflow where proposer models generate candidate responses and aggregator models synthesize higher-quality answers.

Sources:

- https://arxiv.org/abs/2406.04692
- https://www.together.ai/blog/together-moa

Product implication:

- MoA is strongest for high-cognition generation, synthesis, critique, and answer refinement.
- It is weaker as a direct replacement for an agentic tool executor because parallel executors can create conflicting filesystem, network, or permission side effects.

### Hermes Agent MoA

Hermes Agent exposes MoA as an OpenAI-compatible endpoint where multiple models can be configured as a council with an aggregator. Their docs explicitly position it as model orchestration over a mixed set of providers/models.

Sources:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/mixture-of-agents
- https://github.com/NousResearch/hermes-agent

Product implication:

- A practical product surface is not "choose one model". It is "choose a quality mode" that internally maps to proposer/reviewer/aggregator roles.
- Cost and latency must be visible because MoA multiplies calls.
- Hermes' most important MoA design choice is that the aggregator remains the acting model that writes the assistant response and emits tool calls. Reference models run first without tool schemas and provide private analysis to the aggregator. Agent Pi should copy this single-actor design.

### Hermes Agent Broader Architecture

Hermes Agent is useful to study beyond MoA because its product thesis is "the agent grows with you." Its official docs and repository show a long-lived agent with memory, skills, cross-session recall, tool/search progressive disclosure, subagents, scheduling, browser automation, MCP, provider failover, and multiple user surfaces.

Sources:

- https://hermes-agent.nousresearch.com/docs/
- https://github.com/NousResearch/hermes-agent
- https://github.com/NousResearch/hermes-agent/blob/main/AGENTS.md
- https://github.com/NousResearch/hermes-agent/blob/main/website/docs/developer-guide/architecture.md

Product implication:

- Hermes treats learning and context management as first-class runtime systems, not just prompt text.
- Its best lessons for Agent Pi 1.1.3 are not UI duplication. They are control-plane patterns: virtual model providers, progressive disclosure, background maintenance, isolated delegation, and visible resilience.

#### Borrowable Pattern 1: MoA As A Virtual Provider

Hermes makes MoA presets selectable wherever a normal model is selectable. A preset has reference models plus one aggregator; the aggregator is the acting model, receives normal tools, and produces the real response. Reference models receive only conversation text, not the full tool schema or tool transcript.

Source:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/mixture-of-agents

Agent Pi recommendation:

- Implement `qualityOrchestration` as a virtual quality provider internally, but keep the user-facing surface as modes: Fast, Balanced, Deep Quality, Strict Work.
- Keep reference/reviewer models read-only. Only the main executor or aggregator may call tools.
- Do not inject all tool schemas into every reference call; pass bounded objective, task contract, final answer, artifact evidence, and criteria instead.

#### Borrowable Pattern 2: Progressive Disclosure For Tools

Hermes Tool Search replaces many MCP/plugin tools with three bridge tools: search, describe, call. It activates automatically when deferred tool schemas would exceed a threshold of the context window. Core tools remain direct; only deferrable tools are hidden behind search.

Source:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/tool-search

Agent Pi recommendation:

- This is highly relevant to Agent Pi because source/MCP growth can silently consume context and make weak models worse.
- 1.1.3 should not try to fully clone Tool Search, but should add a design target: source/tool schema progressive disclosure for non-core MCP sources.
- The short-term win is an audit metric: count enabled source tools and estimated schema tokens; warn or auto-suggest source narrowing when tool load is high.

#### Borrowable Pattern 3: Skill Lifecycle And Curator

Hermes skills are progressive-disclosure knowledge packages. Its Curator performs deterministic stale/archive transitions and can optionally run an auxiliary-model consolidation pass to patch or merge drifted skills. It never auto-deletes; archive is recoverable.

Sources:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/skills
- https://hermes-agent.nousresearch.com/docs/user-guide/features/curator

Agent Pi recommendation:

- Project Memory Lite already records trusted audit/output entries, but it needs a maintenance layer before it becomes noisy.
- Add a future `Project Memory Curator`: deterministic stale/archive first, optional LLM consolidation later, recoverable archive only.
- This should be project-local and inspectable, matching Agent Pi's current `.agent-pi/brain` design.

#### Borrowable Pattern 4: Bounded Memory Plus External Providers

Hermes built-in memory is intentionally small, curated, and injected at session start. External memory providers are additive and can prefetch relevant memories, sync turns, extract memories on session end, and expose provider-specific tools.

Sources:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/memory
- https://hermes-agent.nousresearch.com/docs/user-guide/features/memory-providers

Agent Pi recommendation:

- Keep Project Memory Lite bounded and trusted. Do not convert it into an unbounded RAG dump.
- For enterprise/private deployment, separate three layers:
  - project-local memory: verified outputs, decisions, gaps
  - company knowledge sources: explicit enabled source/MCP
  - personal preferences: workspace/user profile
- Record quality-orchestration outcomes into project memory only when they are compact and useful for future audits.

#### Borrowable Pattern 5: Isolated Subagents

Hermes `delegate_task` spawns child agents with fresh context, restricted toolsets, and separate terminal sessions. Only final summaries return to the parent context. Its own docs warn that subagents know nothing unless the parent passes explicit context, and that subagent claims must still be verified.

Sources:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/delegation
- https://hermes-agent.nousresearch.com/docs/guides/delegation-patterns

Agent Pi recommendation:

- Agent Pi already has `spawn_session`; for 1.1.3 it should be treated as the safe path for parallel research/review, not parallel file mutation.
- Add a "review-only spawned session" pattern for high-risk artifacts: spawned reviewers get objective, task contract, artifact preview, and no write permissions.
- Delay true background subagents until lifecycle, cancellation, and result re-entry are productized.

#### Borrowable Pattern 6: LSP Diagnostics As Post-Edit Feedback

Hermes runs language servers after file writes and reports newly introduced diagnostics back to the agent. It uses a baseline-before-edit and re-query-after-edit pattern so the agent sees errors introduced by its own change, not the whole existing project backlog.

Source:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/lsp

Agent Pi recommendation:

- This is outside pure MoA but directly improves artifact quality.
- Add to the 1.1.x roadmap as a coding-quality gate: after edit, capture syntax/type diagnostics and feed only introduced errors into Goal Loop evidence.
- Start with TypeScript in this monorepo and Python/PowerShell later.

#### Borrowable Pattern 7: Provider Resilience And Provenance

Hermes has same-provider credential pools, cross-provider fallback, and separate auxiliary-task provider resolution. This is important for multi-model quality orchestration because reviewer, aggregator, compression, and main execution calls may fail independently.

Sources:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/fallback-providers
- https://hermes-agent.nousresearch.com/docs/user-guide/features/credential-pools
- https://hermes-agent.nousresearch.com/docs/integrations/providers

Agent Pi recommendation:

- 1.1.3 should at least persist which model/provider handled each quality role.
- Do not silently pass a quality gate if the reviewer fell back to a weaker or different model. Record fallback in audit evidence.
- Full credential pools can wait; transparent provenance for reviewer and aggregator calls should ship early.

#### Borrowable Pattern 8: Context Files With Progressive Discovery

Hermes loads project context files with priority and progressively discovers subdirectory `AGENTS.md` files when files in those directories become relevant. This reduces startup bloat and preserves prompt-cache stability.

Source:

- https://hermes-agent.nousresearch.com/docs/user-guide/features/context-files

Agent Pi recommendation:

- Agent Pi already handles skill/source/file mention context. For 1.1.3 quality work, add the principle that task-specific rules should be loaded only when evidence shows they are relevant.
- This is especially useful for large enterprise repositories where a global context dump will hurt smaller/private models.

### Sakana AI Fugu

Sakana AI describes Fugu as an LLM orchestration model family that can route prompts to suitable models. This is closer to a learned router than classic MoA: choose the best available model or mixture for a task rather than always running a fixed panel.

Source:

- https://sakana.ai/fugu/

Product implication:

- Agent Pi should start with a deterministic router using task type, risk, cost, context, and local model capabilities.
- A learned/bandit router can come later after telemetry exists.

### Self-Correction Research

Self-Refine and Reflexion both support the same product intuition: models improve when they critique their own output and run another bounded refinement pass. The important engineering detail is that feedback must be structured and task-specific. A vague "improve this" loop is unreliable and can drift.

Sources:

- https://arxiv.org/abs/2303.17651
- https://arxiv.org/abs/2303.11366

Product implication:

- Agent Pi's existing Goal Loop is already aligned with this research: objective, criteria, audit evidence, corrective prompt, bounded retry.
- The 1.1.3 improvement should upgrade the reviewer and corrective prompt generation, not replace the Goal Loop.

## Current Codebase Fit

The codebase already has the right insertion points.

- `packages/server-core/src/sessions/SessionManager.ts`
  - `onProcessingStopped()` runs the goal audit before final UI `complete`.
  - `buildGoalReviewer()` currently calls `managed.agent!.runMiniCompletion(...)`.
  - `maybeInitializeGoalStateForUserMessage()` creates objective, criteria, task contract, budget, and mode.

- `packages/server-core/src/sessions/goal-controller.ts`
  - Performs deterministic file/tool/content checks first.
  - Calls an optional reviewer only when deterministic checks cannot prove success.
  - Builds a hidden corrective prompt and returns `continue` for `auto_improve` / `strict_work`.

- `packages/shared/src/agent/base-agent.ts`
  - `queryLlm(request)` already supports model, system prompt, max tokens, temperature, and structured output.
  - `runMiniCompletion(prompt)` already provides a cheap reviewer path.
  - `preExecuteCallLlm()` shows a reusable pattern for secondary read-only LLM calls.

- `packages/shared/src/sessions/types.ts`
  - `SessionGoalState`, `SessionGoalAuditResult`, `SessionTaskContract`, and budgets are already persisted.

The current gap is not a missing self-correction loop. The gap is that review is mostly a single mini-completion from the active agent. There is no explicit multi-model policy, no council roles, no model performance memory, and no routing logic for when to spend extra calls.

Hermes adds two more gaps worth tracking:

- no progressive disclosure layer for large MCP/source tool surfaces
- no curator for accumulated project-memory and agent-created guidance

## Recommended 1.1.3 Architecture

### Quality Orchestrator

Add a session-layer orchestrator above providers:

```ts
interface QualityOrchestrationPolicy {
  mode: 'off' | 'single_reviewer' | 'multi_reviewer' | 'moa_propose_review' | 'strict_council'
  maxExtraCalls: number
  maxWallClockMs: number
  roles: QualityModelRole[]
}

interface QualityModelRole {
  role: 'router' | 'planner' | 'reviewer' | 'domain_reviewer' | 'aggregator'
  connectionSlug?: string
  model?: string
  temperature?: number
  maxTokens?: number
}
```

Initial location:

- `packages/server-core/src/sessions/quality-orchestrator.ts`
- Called from `SessionManager.buildGoalReviewer()` or directly inside the Goal Loop review path.

### First Slice: Multi-Reviewer, Single Executor

This is the lowest-risk MoA-inspired feature.

Flow:

1. Main executor finishes one turn.
2. Goal Loop runs deterministic checks.
3. If uncertain, Quality Orchestrator asks 2-3 read-only reviewer roles:
   - acceptance reviewer: did it satisfy explicit criteria?
   - artifact reviewer: does the output file/content match the requested deliverable?
   - risk reviewer: what is likely wrong or shallow?
4. Aggregator merges reviewer findings into one `GoalReviewResult`.
5. Goal Loop either passes, continues with a corrective prompt, or stops as needs-review.

Why this first:

- No multi-agent side effects.
- Fits the current `GoalReviewResult` contract.
- Immediately improves self-correction quality.
- Works even with weaker/private models because reviewers can specialize in narrower checks.

### Second Slice: MoA Proposal Before Execution

For high-quality writing, document analysis, research plans, and complex code plans:

1. Router detects high-value task.
2. Several proposer models generate plans, outlines, acceptance criteria, or solution approaches.
3. Aggregator builds a single task contract or execution plan.
4. One executor performs tool/file work.
5. Goal Loop audits the artifact.

This should not run for casual chat or small tasks.

### Third Slice: Fugu-Style Router

Start with deterministic routing:

- task type: code, document, data, research, automation
- risk: file write, external API, financial/legal claims, enterprise output
- required artifact: PDF/Word/Excel/code/test
- local model capability: context window, tool support, structured output support
- budget: user mode, workspace setting, max wall clock, max extra calls

Later, persist outcomes in Project Memory Lite or workspace metadata:

- reviewer pass/fail rate by model and task type
- number of retries before pass
- user accepted / manually corrected / reopened
- artifact quality score
- cost and latency
- Implementation note: Project Memory Lite now stores quality reviewer facts with task type and surfaces compact task/role aggregates in reviewer performance summaries, including pass/fail/uncertain counts, common gaps, and average latency.
- Implementation note: the first deterministic router now builds an inspectable quality route for council review, assigns role-specific reviewer models from workspace `goalLoop.reviewerModels`, and records `quality_route` evidence with task, roles, model assignments, and matching telemetry gaps.
- Implementation note: Project Memory Lite now also persists `quality_route_fact` outcomes to `.agent-pi/brain/facts.jsonl`, tying the route choice to the final audit status, failure categories, model assignments, telemetry roles, and common gaps.
- Implementation note: route outcome facts are now aggregated back into reviewer performance memory and consumed by the deterministic router as `route_history`, giving future learned routing an audited pass/fail history for the current task type.
- Implementation note: learned quality telemetry can now be reset without deleting other project facts. The reset removes only `quality_reviewer_fact` and `quality_route_fact` from `facts.jsonl`, is exposed through a session RPC/Electron API, and is available from the Session Info Project Memory row.
- Implementation note: conservative learned routing now reacts to degraded local route history by adding a read-only `route_history_reviewer` focused on repeated historical gaps. It does not automatically switch models or remove existing reviewers.
- Implementation note: workspace `goalLoop.maxExtraReviewers` now caps learned-routing add-on reviewers. The default is 1, while 0 keeps `route_health` auditable but disables the extra `route_history_reviewer` call.

Then add a bandit-style router that improves model selection from local outcomes.

### Hermes-Derived Additions

These should be considered adjacent 1.1.3 work because they increase the probability that the agent produces acceptable artifacts:

1. Quality-call provenance
   - Persist role, provider, model, fallback status, latency, and token usage for reviewer/proposer/aggregator calls.
   - Feed this into the Goal Loop audit history and UI.

2. Source/tool load audit
   - Count enabled MCP/API/source tools and estimate schema pressure.
   - Warn or suggest narrowing when the active model is likely overloaded by the tool surface.

3. Review-only spawned sessions
   - Use existing `spawn_session` infrastructure for isolated review, but start with read-only or low-permission reviewer sessions.
   - Only summaries and structured findings re-enter the parent session.

4. Project Memory Curator design
   - Deterministic stale/archive first.
   - Optional LLM consolidation later.
   - Recoverable archive, never silent deletion.

5. Coding artifact diagnostics
   - Feed introduced syntax/type diagnostics into Goal Loop evidence after edits.
   - Start narrow; do not block non-code tasks.
   - Implementation note: code task verification failures such as typecheck/test/build/lint diagnostics now enter Goal Loop evidence and can trigger one more automatic improvement pass, while unrelated tool failures still stop for review.

## Private Deployment Strategy

For enterprises using weaker domestic or local models, the goal should not be "small model magically equals Opus". A more realistic and valuable goal is:

- narrow each model's role
- make each role evidence-bound
- make failures observable
- retry only with concrete missing criteria
- escalate only when configured

Recommended private stack:

- local small model: router/classifier
- local small or medium model: first reviewer
- domain-tuned local model: document/domain reviewer
- strongest available private model: aggregator or executor only when needed
- deterministic checks: file existence, format, citations, tests, table/schema validation

This makes weak models useful because they no longer need to hold the whole task in one pass. They can contribute bounded judgments to the Goal Loop.

## Product Surface

Do not expose "MoA" as the main user concept in v1.1.3. Expose quality modes:

- Fast: current behavior or single reviewer.
- Balanced: Goal Loop plus one reviewer.
- Deep Quality: multi-reviewer council and artifact-aware review.
- Strict Work: multi-reviewer council, stronger artifact checks, lower pass tolerance.

Workspace settings:

- `qualityOrchestration.defaultMode`
- `qualityOrchestration.maxExtraCalls`
- `qualityOrchestration.maxWallClockMs`
- `qualityOrchestration.reviewerModels`
- `qualityOrchestration.aggregatorModel`
- `qualityOrchestration.allowCrossConnectionModels`
- `qualityOrchestration.maxExtraReviewers`

Session UI:

- show latest audit summary
- show which quality mode ran
- show number of extra model calls
- show "why another pass is running"
- keep manual controls: accept as done, run one more improvement, turn off

## Implementation Plan

### Phase 1: Council Reviewer

Success criteria:

- Existing Goal Loop behavior remains unchanged when quality orchestration is off.
- In `multi_reviewer` mode, the reviewer runs several bounded read-only `queryLlm()` calls.
- Aggregator returns the existing `GoalReviewResult` shape.
- Reviewer/proposer/aggregator role provenance is recorded in audit evidence.
- Same missing criteria twice still stops instead of looping.
- Tests cover pass, fail, uncertain, timeout, malformed reviewer JSON, and budget exhaustion.
- Implementation note: the council now keeps the three base read-only reviewers, and adds task-contract-specific reviewers for research source grounding and code implementation verification.

Suggested tests:

- `packages/server-core/src/sessions/quality-orchestrator.test.ts`
- Extend `packages/server-core/src/sessions/goal-loop-session-manager.test.ts`

### Phase 2: Task Contract MoA

Success criteria:

- For high-quality document/research/code-plan tasks, proposer models can improve the task contract before execution.
- Only one executor writes files or invokes side-effecting tools.
- The final task contract is persisted in `SessionGoalState.taskContract`.
- Implementation note: the persisted task contract now also becomes a bounded volatile `<goal_contract>` execution context before the agent acts, so deliverables, must-preserve requirements, evidence needs, acceptance criteria, and forbidden shortcuts are visible before the first completion attempt instead of only during audit/review.

### Phase 2b: Hermes-Style Context And Tool Pressure Controls

Success criteria:

- Session metadata can report approximate enabled-tool/schema pressure.
- The UI can surface a non-blocking warning when too many source tools are active for the selected model.
- No behavior changes for small toolsets.
- Future Tool Search can be implemented without changing provider backends.
- Implementation note: source/tool pressure is now calculated in shared session logic from enabled sources, selected model context window, and current input tokens. The session popover shows a non-blocking row only when thresholds are crossed. PromptBuilder also injects a volatile `<context_pressure>` strategy block before execution when many sources are enabled, and the Goal Loop records context pressure as audit evidence plus a narrowing checkpoint when an auto-improvement pass is already needed.

### Phase 3: Model Router And Telemetry

Success criteria:

- Router chooses mode/models from task type, budget, and configured capabilities.
- Outcomes are recorded as inspectable local JSONL.
- Router can be disabled and has deterministic fallback.
- Implementation note: reviewer telemetry summaries now include task/role aggregates plus recent facts, so future routing can prefer reviewers by observed local performance instead of relying only on static model configuration.
- Implementation note: council review now routes through a deterministic quality route before making reviewer calls. The route is auditable in goal evidence and can apply per-role model overrides, giving private deployments a practical first step toward MoA without changing the default setup.
- Implementation note: route choices are now converted from audit evidence into structured Project Memory facts, closing the first telemetry loop needed for learned routing.

### Phase 3b: Project Memory Curator

Success criteria:

- Project Memory Lite entries can be marked active, stale, or archived.
- Archival is deterministic and recoverable.
- Optional LLM consolidation is off by default.
- Curator output is inspectable as a report under `.agent-pi/brain`.
- Implementation note: prompt-time Project Memory context now performs deterministic filtering before injection: archived, stale, and unverified entries are excluded, while verified/user-promoted entries and actionable needs-review gaps remain eligible. This is the first Curator step and does not delete or rewrite stored memory.

### Phase 4: Learned Routing

Success criteria:

- Routing improves from local historical outcomes without sending private data externally.
- Workspace owner can reset telemetry.
- The system can explain why a model/council was selected.
- Implementation note: the first learned-routing substrate is now in place: route outcomes are persisted, summarized by task/route/model assignment, and read back into route evidence as historical counts. Automatic route switching is still intentionally not enabled until enough samples and override controls exist.
- Implementation note: the reset control is implemented at Project Memory, session API, Electron API, and Session Info UI level, so bad learned-routing history can be cleared before automatic route switching is enabled.
- Implementation note: if a route has enough local samples and failures dominate, the router marks `route_health=degraded` and adds `route_history_reviewer` to force one more bounded review pass against repeated failure patterns.
- Implementation note: extra learned-routing reviewers are budget-controlled by workspace `goalLoop.maxExtraReviewers`; disabling them does not hide degraded route evidence from audit logs.

## Non-Goals For 1.1.3

- Do not run multiple full tool-using executors against the same workspace.
- Do not train or fine-tune a router before there is local outcome data.
- Do not promise Opus-level quality from cheap models across all tasks.
- Do not hide extra model calls from the user.
- Do not replace deterministic artifact checks with LLM judges.
- Do not clone Hermes' whole gateway/messaging/runtime platform.
- Do not ship full background subagent orchestration before cancellation, permissions, and result re-entry are explicit.

## Key Risks

- Cost and latency: cap extra calls and show them in UI.
- Reviewer hallucination: deterministic checks and artifact previews must come first.
- Collusion or shared blind spots: use diverse prompts/roles, not just three copies of the same judge.
- Context leakage in private deployments: allow local-only reviewer profiles.
- Scope drift during retries: corrective prompts must preserve the persisted task contract.
- Provider mismatch: cross-connection querying needs a server-level LLM query service; same-connection model switching is easier and should ship first.

## Final Recommendation

Make 1.1.3 a quality-control release:

1. Upgrade the existing Goal Loop from single mini-review to model-council review.
2. Keep side effects single-executor.
3. Add MoA-style proposer/aggregator only for planning, review, and artifact refinement.
4. Copy Hermes' virtual-provider idea internally, but expose it as quality modes rather than raw MoA plumbing.
5. Add provenance for every quality role so users can see which model judged what.
6. Start measuring source/tool pressure; plan Tool Search-style progressive disclosure for large MCP/source surfaces.
7. Add a Project Memory Curator design before memory grows noisy.
8. Build deterministic Fugu-style routing first, learned routing later.
9. Treat private/local models as narrow specialists inside the loop, not as direct replacements for top closed models.

This direction directly serves the product goal: improve whether the final artifact meets the user's expected outcome, and make the agent notice and correct its own drift before the user has to intervene.
